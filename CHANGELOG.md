# Changelog

Pre-release. Work is tracked by milestone (Mn) following the roadmap in `DESIGN.md` §4.10.
Nothing is published yet; all versions are `0.1.0`.

## Unreleased

- M296: A/V `flvmux` (`FlvMuxN`), the last muxer that was single-track only, so every standard container (`mpegtsmux`/`mp4mux`/`matroskamux`/`flvmux`) now does A/V: `videotestsrc ! videoconvert ! x264enc ! m.   audiotestsrc ! avenc_aac ! m.   flvmux name=m ! filesink location=av.flv`. A multi-input muxer registered as the `flvmux` fan-in (the parser picks it for >1 input link, the single-track `FlvMux` for one), FLV's one-video + one-audio model (H.264 + AAC); a second pad of either kind is rejected. Unlike the single-track muxer (media frames only), it writes the decoder-config sequence-header tags a player needs: `FlvMuxer` gained an A/V path (`new_av`, `push_video`/`push_audio`) that emits the H.264 `avcC` record and AAC `AudioSpecificConfig` as FLV AVC/AAC sequence headers up front, captured in-band from the first AU (parameter sets from the first IDR, ASC from the first ADTS header), then interleaves video / audio media tags by PTS with the FLV keyframe flag set per frame. Samples are reframed to FLV's convention (video AVCC length-prefixed, audio de-ADTS'd). Verified end-to-end with ffprobe + a full ffmpeg decode (h264 320x240 + aac 48 kHz stereo); integration-tested (`m296`). std-gated like `mkvmuxn` (reuses the MP4 family's NAL / ADTS / avcC helpers); the single-track baseline `flvmux` is unchanged.
- M295: remove the `Mp4Sink` convenience element (no combined muxer+file-writer elements: muxing is a separate element feeding any sink, the gst model `mp4mux ! filesink`). g2g muxes *fragmented* MP4 (forward-only, no moov backpatching), so a dedicated seekable sink was never needed; `Mp4Sink` predated the M291 muxer/sink split and only duplicated `Mp4Mux`. Its module `mp4sink.rs` is renamed `fmp4mux.rs` and keeps just the pure `Fmp4Muxer` box writer + the NAL / `avcC` / `hvcC` helpers (reused by `mp4mux`/`mp4muxn`/`mkvmuxn`); the element, its file I/O, and the dead `fragments()` accessor are gone. The 6 tests that used `Mp4Sink` as an fMP4 fixture now generate the bytes through `Mp4Mux` (`m28`/`m31` still write a file for `Mp4Src` to read; `m156`/`m158`/`m160` capture the forwarded bytes directly); the standalone `m24_mp4sink` is dropped (its round-trip / `ftyp`+`moov` / fragment-per-AU coverage is `m291` + the `fmp4mux` unit tests, and its reject-caps check moved into `m291`).
- M294: A/V `matroskamux` (`MkvMuxN`), so a Matroska / WebM recording carries video and audio together: `videotestsrc ! videoconvert ! x264enc ! m.   audiotestsrc ! avenc_aac ! m.   matroskamux name=m ! filesink location=av.mkv`. A multi-input muxer (the A/V analog of the single-track `MkvMux`, registered as the `matroskamux` fan-in the way `mpegtsmux`/`mp4mux` are, so >1 input link picks it): H.264/H.265 video + AAC audio, access units interleaved by PTS via the M204 `InputAggregator` into shared-timeline Clusters. `MatroskaMuxer` gained a multi-track path (`new_multi`, `push_frame_on`): the Tracks element carries one TrackEntry per stream, each with its `CodecPrivate` (the avcC / hvcC record for H.26x, the AAC AudioSpecificConfig), built once each track's init arrives in-band from the first AU (video parameter sets ride the first IDR; the ASC is synthesised from the first ADTS header). Samples are reframed to what the Matroska codec mappings expect: video AVCC length-prefixed, audio de-ADTS'd. Verified end-to-end with ffprobe + a full ffmpeg decode (h264 320x240 + aac 48 kHz stereo, both tracks decode cleanly); integration-tested (`m294`). Reuses the MP4 family's NAL / ADTS / avcC helpers, so the fan-in is std-gated (the single-track baseline `matroskamux` is unchanged).
- M293: A/V `mp4mux` (`Mp4MuxN`), so a fragmented-MP4 recording carries video and audio together: `videotestsrc ! videoconvert ! x264enc ! mp4mux name=m   audiotestsrc ! avenc_aac ! m.   m. ! filesink location=av.mp4`. A multi-input muxer (the A/V analog of the single-track `Mp4Mux`, registered as the `mp4mux` fan-in muxer the way `mpegtsmux`/`tsmuxn` is, so >1 input link picks it): H.264/H.265 video + AAC audio, access units interleaved by PTS via the M204 `InputAggregator`. The `moov` carries one `trak` per stream and is built once each track's init data arrives in-band (video parameter sets ride the first IDR; the audio AudioSpecificConfig is synthesised from the first ADTS header, with the ADTS stripped from the `mdat` samples); then one `moof`+`mdat` per AU with a per-track `tfdt`. Shared box helpers extracted from `mp4sink`/`mp4audiosink`. Verified end-to-end with ffprobe + a full ffmpeg decode (h264 320x240 + aac 48 kHz stereo, both tracks decode); integration-tested (`m293`).
- M292: `avenc_aac` (`aacenc`) libavcodec AAC-LC audio encoder, the audio companion of the H.264 `ffmpegenc` and the Linux audio-encode path the A/V muxers needed (the existing `MfAacEncode` is Windows-only). Interleaved PCM in (`PcmS16Le`/`PcmF32Le`), ADTS-framed AAC out (`Caps::Audio{Aac}`, the elementary-stream convention `aacparse` and the muxers expect). Converts PCM to the encoder's planar-float format and buffers into 1024-sample AAC frames, flushing the tail at EOS; allocates the codec context with `Context::new_with_codec` (the M289 fix). Verified `audiotestsrc ! avenc_aac ! aacparse ! fakesink` and an ffprobe-valid `.aac` (48 kHz stereo). Also unlocks AAC audio for `mpegtsmux`/`matroskamux`/`flvmux`, which already accept it.
- M291: `mp4mux` element (gst `mp4mux`/`qtmux` analog), so `... ! x264enc ! mp4mux ! filesink location=out.mp4` works (gst has no `mp4sink`; muxing is a separate element feeding any sink). H.264/H.265 Annex-B in, an ISO-BMFF byte stream (`Caps::ByteStream{IsoBmff}`) forwarded downstream: `ftyp`+`moov` init segment then one `moof`+`mdat` fragment per access unit. Extracted the fragmented-MP4 box writer out of `Mp4Sink` into a shared pure `Fmp4Muxer`; `Mp4Sink` (writes to a file) and `Mp4Mux` (forwards downstream) now wrap the same writer, mirroring `MkvMux`/`MatroskaMuxer`. Registered `mp4mux` + the `qtmux` alias (std-gated, like its module). Round-trip tested (`m291`: mux through `Mp4Mux`, recover byte-exact AUs through `Fmp4Demux`); ffprobe confirms a real playable MP4.
- M290: make `videorate` caps-driven, the last rate/format-changing element that wasn't. The standard gst idiom `videorate ! video/x-raw,framerate=N` previously failed `CapsMismatch` because `videorate` ignored the downstream pin and forced its own 30 fps default; only the `framerate=` property worked. New `VideoRate::auto()` (now the registry default) takes the target framerate from the negotiated output caps via a `DerivedCoupled` constraint (format + geometry passthrough, framerate the retargeted field), matching M186/M187 `videoconvert`/`videoscale`/`audioresample`. `videorate framerate=N` still works (sets the property, disables auto); bare `videorate` passes the input rate through. Unit + smoke verified up/downsample.
- M289: fix `x264enc` (libx264 via `FfmpegH264Enc`) failing to open at runtime. The encoder context was allocated with `encoder::new()` (no codec), leaving the generic legacy `AVCodecContext` defaults (`qmin=2`, `qmax=31`, `max_qdiff=3`, `qcompress=0.5`, `me_range=0`) which are exactly libx264's "broken ffmpeg default settings" fingerprint, so it aborted the open (`Hardware(Other)`) despite the `preset` being passed. Now allocates with `Context::new_with_codec(codec)` so codec-appropriate defaults apply, matching the `ffmpeg` CLI. Verified `videotestsrc ! videoconvert ! x264enc ! fakesink` encodes end-to-end. Also gated the `gst_compat` `unknown_encoder_gets_a_suggestion` test to `not(feature = "ffmpeg")` (it asserts `x264enc` is unregistered, false under the feature).
- M288: fix the stacked auto-converter cascade bug. Two chained auto `videoconvert`s into a format pin (`videotestsrc ! videoconvert ! videoconvert ! NV12`) emitted the raw RGBA bytes mislabeled NV12. The transform arm calls `configure_pipeline(input)` then `process(CapsChanged(forward_caps))` with the element's pre-fixed *output*, but `videoconvert` / `videoscale` / `audioconvert` / `audioresample` re-read that arg as a new *input* (`accept_input`), clobbering their input to the output format so the next frame became an unconverted `X->X` passthrough. They now forward the arg downstream and record `last_caps` (matching the decoder elements), leaving the `configure_pipeline`-set input alone. Byte-compare regression test `m288_stacked_convert_bytes` (1 vs 2 converts must produce identical NV12). Contract documented in DESIGN.md §4.13.4.
- M287: end-of-run telemetry report. New `RunStats::report()` formats the run telemetry, frame counts + drop rate %, the aggregated declared latency window (`min .. max`, live/non-live, the per-element `latency()` fold the runner already computes), the elected clock, and the negotiated head allocation, into a readable block. `g2g-launch` prints it at end (replacing the terse one-liner) together with the measured wall-clock throughput (e.g. `0.03 s wall, 601.8 fps`). The latency is *declared*, not a measured runtime histogram; per-element / per-link p50/p99 + fill-level needs frame-level instrumentation in the runner and stays a follow-up. Unit-tested formatting (drop-rate computation, bounded vs unbounded latency window).
- M286: runner-loop throughput bench + a CI regression guard. New `runner` bench in `g2g-bench` (criterion `async_tokio` + tokio) times the bounded per-edge channel, the runner loop's inner transport (the full `run_graph` paces to PTS, so it is unsuitable for a microbench), sweeping link capacity 2 (live) vs 8 (throughput) to show the backpressure round-trip cost; ~350K frames/s at cap 2 here. New `.github/workflows/bench.yml` (separate from the lean main CI, so criterion never hits the check/test/clippy jobs; runs only when `g2g-core` / `g2g-plugins` / `g2g-bench` change or on dispatch) benches the PR head and its base, then `bench_compare.py` fails the job if any benchmark's mean regressed > 50% vs base, a deliberately loose threshold given shared-runner noise (catches a lost fast path / accidental O(n^2), not drift). Comparator unit-checked locally both ways.
- M285: per-edge memory domain in the DOT dump. New `AsyncElement::output_memory` / `SourceLoop::output_memory` (default `MemoryDomainKind::System`, mirrored on the dyn traits + `GraphNodeRef::output_memory`) declares the domain of the frames an element emits, the runtime peer of M276's auto-plug `ElementDesc::output_memory` (it is not part of `Caps`). `NvDec` overrides it to `Cuda` (it emits NV12 in VRAM). `negotiate_graph` now also returns the per-edge memory domain (the producing node's `output_memory`), and `g2g-launch --dot` passes it so a GPU / zero-copy link is drawn bold + labelled `memory:Cuda` while CPU links stay plain. Fan-in / fan-out nodes report `System` (per-edge propagation through them is deferred). Tested: `m282_dot_negotiated_caps` asserts a CPU pipeline's edges are all `System` and the dump carries no `memory:` marking.
- M284: criterion benchmarks for the hot paths, so the latency moat is regression-guarded. A standalone `g2g-bench` crate (excluded from the workspace, like `examples/g2g-size` and the bevy demo, because criterion pulls plotters / rayon that `--all-targets` would otherwise build on every CI job, `required-features` does not gate dev-deps there) holds two benches: `caps` (the negotiation hot path, the hardest code) times `Caps::intersect` / `CapsSet::intersect`+`fixate` / `solve_linear` / `solve_graph` on a decode chain; `convert` times the per-pixel `videoconvert::convert` (now `pub` so the bench reaches it) for RGBA<->NV12/I420 + NV12<->I420 at 1080p with a pixels/s throughput. New `cargo xtask bench` runs them by manifest path (criterion args pass through, e.g. `-- --save-baseline main`). Indicative on the dev host: caps intersect ~13 ns, `solve_graph` ~1.6 us, RGBA->NV12 ~4.1 ms/1080p-frame. CI stays lean (criterion is not in the workspace lock; verified `--workspace --all-targets` builds none of it).
- M283: `xtask ffi-probe <header> <struct> [--field f]...`, automating the hand-rolled-FFI ritual (the C probe + `repr(C)` size assert convention `cuda.rs` / `nvenc.rs` use). It generates a C program that includes the header(s) and prints `sizeof` of the struct plus `offsetof` of each requested field, compiles it (`cc`, or `$CC` / `--cc`, with repeatable `-I` include dirs), runs it, and emits the ready-to-paste `const _: () = assert!(core::mem::size_of::<Struct>() == N);`. This locks down a version-tagged SDK struct's layout before trusting it, and catches SDK drift (a changed size fails the build, not the GPU). Verified on this host against `ffnvcodec/nvEncodeAPI.h`: `NV_ENC_INITIALIZE_PARAMS` probes to 1800 bytes, matching the committed assert in `nvenc.rs`. The C-source generation and size parsing are unit-tested.
- M282: negotiated caps in the DOT dump, closing the M279 follow-up. New `negotiate_graph` runs startup caps negotiation only (Phase 1 source-caps probe + Phase 2 whole-graph solve) without running the pipeline, returning the `ValidatedGraph` plus the fixated `Caps` per edge (indexed by edge id, as `DotAnnotations::edge_caps` expects). The runner's Phase 2 is refactored onto two shared helpers it now also uses (`build_node_constraints`, `caps_label`), so there is one constraint-builder, not two. `g2g-launch --dot` negotiates first and renders the chosen caps on each edge (e.g. `video/x-raw,format=RGBA,width=320,height=240,framerate=30/1`); on a negotiation failure it warns and falls back to a topology-only dump (re-parsing, since negotiation consumed the graph). Note: `--dot` now probes source caps, so a source that connects on `intercept_caps` (a live ingress) does so for the dump too. Tested: `m282_dot_negotiated_caps` (caps land on both edges of a 3-element chain; a video->audio pin fails to negotiate rather than panicking).
- M281: `xtask` dev-command crate (`cargo xtask <cmd>` via a `.cargo/config.toml` alias), the third developer-tooling item, consolidating the build / test invocations that were shell-history tribal knowledge. Zero dependencies (it only orchestrates `cargo` + toolchain tools). Commands: `ci` runs locally what CI runs (workspace check / test / clippy, the Linux feature build of g2g-plugins + g2g-ml, the embassy no-alloc tests, and the wasm core check), `--locked` like CI, stopping at the first failure; `test --here` probes the host (NVIDIA via `nvidia-smi`, VAAPI / Opus / ALSA / Pulse / Wayland / ffmpeg via `pkg-config`, `/dev/video*` cameras, `/dev/dri` GPU nodes) and runs exactly the feature-gated tests it supports (`--dry-run` prints the detected plan only); `size` builds the `examples/g2g-size` Cortex-M harness and reports the gc-sectioned `.text` (locating `rust-lld` in the toolchain sysroot, 3280 bytes here); `wasm` builds the wasm32 targets (core `runtime`, plugins `web` / `web-codecs`). The cross builds prepend `~/.cargo/bin` to `PATH` so cargo uses the rustup toolchain, baking in the Fedora distro-`rustc` gotcha; `web-codecs` gets `--cfg=web_sys_unstable_apis`. Unit-tested capability-to-feature mapping; `size` / `wasm` / dry-run probe validated on the dev host.
- M280: caps-negotiation explainer, the second developer-tooling item. `solve_graph` now narrates the DAG negotiation under a reserved `caps` log category: a setup dump (each node's constraint, e.g. `VideoTestSrc produces video/x-raw,...` / `VideoConvert derives output (coupled)` / `FakeSink accepts ANY`) and, per edge, the surviving `CapsSet` and its fixated result (`VideoTestSrc -> VideoConvert: ... ✓ -> video/x-raw,format=RGBA,...`). On failure it narrates at ERROR: `no caps overlap between <A> and <B>` plus the set on every edge incident to the conflict (so a `CapsMismatch` reads as "these two can't agree, here's what each wanted"), and an unfixable edge logs `... ✗ cannot fixate`. New `solve_graph_labeled` lets the caller supply node labels; the runner passes each element's `log_category` so the narration reads in element names, while direct callers get an `n{id}:{kind}` default. Turned on by `G2G_CAPS_TRACE=1` (a boolean shortcut, or a level name / number for verbosity) or `G2G_DEBUG=caps:debug`; both install the stderr sink via `log::init_from_env`, which `g2g-launch` already calls. All trace formatting is skipped unless the category is enabled (one atomic load when off). Tested: solver unit tests (set / constraint formatting, labeled == default solution) and an `m280_caps_trace` integration test capturing the log to assert the success + failure narration.
- M279: pipeline -> Graphviz DOT visualizer (the `GST_DEBUG_DUMP_DOT_DIR` analog), the first developer-tooling item. New `g2g_core::dot` module adds `Graph::to_dot` / `ValidatedGraph::to_dot`: a `digraph { .. }` with role-coded nodes (green sources, red sinks, blue transforms, diamond tees / trapezium muxers), labelled by a caller-supplied closure (returning `None` falls back to the structural kind, so a tee reads as `tee`). Edges carry the negotiated caps and memory domain from a `DotAnnotations { edge_caps, edge_memory }` (both indexed by edge id, matching `solve_graph`'s `Vec<Caps>`), the non-default `LinkPolicy`, and fan-out / fan-in pad indices; a non-`System` (GPU / zero-copy) link is drawn bold. Self-contained `no_std + alloc` (only formats `String`s, no I/O), builds on `thumbv7em`. Wired into `g2g-launch --dot`: dumps the parsed pipeline to stdout (pipe to `dot -Tsvg`) labelled by each element's log category, without running it; supporting `GraphNodeRef::log_category` plus `Graph::{node_count,node_kind,element}` and `ValidatedGraph::edges` accessors. Tested: `g2g-core` unit tests (linear chain, structural fallback + pads, GPU-memory marking, escaping) and an end-to-end `m279_dot` launch-parser test (linear dump, tee branch pads + leaky-queue policy).
- M278: zero-copy GPU-render egress, the `examples/bevy-g2g-stream` rewire (closes the M271 follow-up). Bevy now renders on g2g's *interop* wgpu device (new `create_interop_device_full`, opened with the adapter's full features so a renderer is happy on it; handed to Bevy via `RenderCreation::Manual`), so a render-world system copies each rendered texture device->device through the `WgpuToCuda` bridge (M275) into a CUDA surface and encodes it with native `NvEnc` (M269/M271), emitting H.264 access units with **no GPU->CPU read-back**; the M267 version read the full RGBA frame back and encoded via ffmpeg. The encoded access units cross to a sink thread (`AppSrc(H.264) -> FileSink | WebRtcSink`) over a channel, so the WHIP egress is unchanged. Supporting changes: `WgpuToCuda::ingest_texture` made public (a host renders into the bridge directly), `InteropDevice.adapter`/`.instance` exposed (Bevy's `RenderResources` needs all four handles). Two NVIDIA-driver hazards handled: `synchronous_pipeline_compilation: true` (Bevy's async Vulkan pipeline compile faults when it overlaps the CUDA encode on the shared driver), and `std::process::exit` after flush (skip the render-world CUDA/wgpu teardown that races Bevy's device shutdown). Validated on the RTX 3060: a run produces a 640x480 H.264 stream that decodes back to the rendered scene (cube on a lit plane). The example is excluded from the workspace / CI as before.
- M277: NvEnc depth, two NVENC encoder improvements. (1) Output-bitstream pool: the output buffer is now created once at session open and reused every frame (NVENC runs sync-mode, one locked + copied output per encode), replacing the per-frame `nvEncCreateBitstreamBuffer` / `nvEncDestroyBitstreamBuffer`; it is destroyed before the encoder on session teardown. (2) Runtime bitrate retarget via `nvEncReconfigureEncoder`: `set_property("bitrate", ..)` (an adaptive sink, e.g. a WebRTC BWE drop) is applied to the live session before the next frame, with no session re-open and no forced IDR (`resetEncoder=0`), so the rate control retargets smoothly. The config / init building is factored into shared helpers so open and reconfigure stay in lock-step. New `NV_ENC_RECONFIGURE_PARAMS` FFI (size-asserted 1816 bytes against nvEncodeAPI.h 13.0; the reconfigure fn pointer is now typed); `nvenc_round_trips_a_cuda_nv12_surface` on the RTX 3060 now drops 4 Mbps -> 1 Mbps mid-stream and decodes the result back, and asserts the applied bitrate tracked the retarget. Remaining NvEnc depth (10-bit / P010, finite-GOP repeatSPSPPS, mid-stream resolution reconfigure) stays deferred; AV1 needs RTX 40-series.
- M276: domain-aware auto-plug. The memory domain a decoder produces (`NvDec` -> NV12 in `MemoryDomain::Cuda`) is not encoded in `Caps` and threading it through all 446 `Caps` sites would be wrong (it is orthogonal to the format algebra), so it rides as a *memory feature* on the auto-plug metadata: `ElementDesc::output_memory: MemoryDomainKind` (GStreamer's `memory:CUDAMemory` caps-feature analog), set per factory by `ElementFactory::produces(kind)` and defaulting to `System`. `find_chain_preferring(.., preferred)` and `Registry::{autoplug,autoplug_names,decodebin}_preferring(.., preferred)` bias ties toward a chain whose terminal element emits `preferred` (still breadth-first, so a shorter chain always wins; among equal-length chains the matching-memory candidate is tried first). The default `System` preference is exactly registration order, so a plain pipeline is unchanged and `NvDec` (registered last, now tagged `produces(Cuda)`) never hijacks a CPU path; a GPU consumer requests `Cuda` and gets `NvDec` ahead of the CPU decoder. New `nvdec`+`ffmpeg` test asserts default H.264 decode picks `ffmpegdec` while a `Cuda` preference picks `nvdec` (no GPU; metadata-only search). Deriving the preference from a downstream consumer's accepted memory is a follow-up. Architecture in DESIGN.md 4.13.9.
- M275: `WgpuToCuda` (M271) promoted to a pipeline `AsyncElement` plus a linear-buffer reuse pool. The bridge now implements `AsyncElement` (`Identity(RGBA)` caps, `MemoryDomain::WgpuTexture` in -> `MemoryDomain::Cuda` out): `process` extracts the upstream texture (via `gpu::texture_of`), copies it into the bridge's exportable render target on the interop device, drains, and emits a CUDA RGBA frame for `NvEnc`, no read-back. Unlike the lazy-device M220 `CudaToWgpu`, the element owns its interop device up front and exposes `device()` / `queue()` so the upstream renderer draws onto it (the `WgpuSink` app-wired pattern; a `wgpu::Texture` is device-bound and only this device's memory is CUDA-importable). `to_cuda_frame` now draws its linear output buffer from a `LinearBufferPool` free list (recycled by size when the emitted frame's keep-alive drops) instead of `cuMemAlloc`/`cuMemFree` per frame. The `gpu` module gate gains `cuda-wgpu` (it reuses `texture_of` / `WgpuTextureKeepAlive`). New on-hardware test `wgpu_to_cuda_element_bridges_and_recycles` drives 10 frames through the element into NVENC on the RTX 3060, each consumed + dropped before the next so the pool recycles.
- M274: registry wiring for the ffmpeg H.264 encoder (`FfmpegH264Enc`, M266), the encode-side gap surfaced during M272. Two launch names, `ffmpegenc` (default NVENC `h264_nvenc` backend) and `x264enc` (libx264 software backend), plus the gst `avenc_h264` alias resolving software-first (`x264enc` -> `ffmpegenc`). Launch-only: an encoder is never an auto-plug candidate. New `ffmpeg`-gated test asserts all three names resolve to constructible elements (no GPU; `new()` opens no libavcodec context).
- M273: HEVC (H.265) for the native `NvEnc` / `NvDec` (M269 / M270), the highest-value codec gap on Ampere/3060 (NVENC AV1 still needs RTX 40-series). The encode/decode paths are codec-agnostic bar the codec id, so the addition is small: `NvEnc` gains a `codec` field / `with_codec(VideoCodec)` builder / `codec` property (`h264` default | `hevc`), picking `NV_ENC_CODEC_HEVC_GUID` for the preset + initialize calls and announcing `CompressedVideo{H265}` out; its `DerivedOutput` and pad-template source now advertise both codecs, and configure rejects anything but H.264 / HEVC. `NvDec` infers the codec from its input caps (accepts `CompressedVideo{H264|H265}`), maps it to the `cudaVideoCodec` (H264=4, HEVC=8) stored on the decoder state, and creates the NVCUVID parser + decoder for it; the rest of the callback machinery is unchanged. New on-hardware test `nvenc_to_nvdec_hevc_round_trip_on_gpu` runs the full native HEVC loop on the RTX 3060 (CUDA NV12 -> `NvEnc(hevc)` -> HEVC Annex-B -> `NvDec` -> CUDA NV12, decoded luma verified), alongside the existing H.264 one; a no-GPU unit test checks the HEVC GUID / H.265 output-caps selection. 10-bit, mid-stream resolution reconfigure, and a finite-GOP `repeatSPSPPS` remain deferred.

- M272: registry / auto-plug wiring for the native `NvEnc` / `NvDec` (M269 / M270), so `gst-launch` / `gst-inspect` / `decodebin` see them. Both register as named launch elements (`nvenc`, `nvdec`) under their feature gates, with GStreamer-canonical aliases resolving to them (`nvh264enc` / `nvv4l2h264enc` -> `nvenc`, `nvh264dec` -> `nvdec`, the same first-registered rule as the VA-API names). `NvDec` is also an auto-plug `ElementFactory` candidate, but **registered last, after the CPU decoders, on purpose**: it emits NV12 in CUDA device memory (`MemoryDomain::Cuda`), which caps do not encode, so a generic `decodebin` to a system-memory sink must still pick a CPU decoder; ranking is by registration order (breadth-first finds the earliest-registered among equal-length chains), so registering `NvDec` last keeps it available to an explicit chain without hijacking CPU pipelines. Selecting the native decoder is therefore explicit (the `nvdec` / `nvh264dec` name) until a caps memory-feature gates auto-plug by domain (the follow-up). New test (`nvenc nvdec` features) asserts the names and aliases resolve to constructible elements and that `nvdec` is an auto-plug factory; it needs no GPU (`new()` opens nothing, the session / context are lazy at configure). Noted gap surfaced while here: `FfmpegH264Enc` (M266) is still unregistered; folding it in (`ffmpegenc` + `x264enc` / `avenc_h264` aliases) is a small separate follow-up.

- M271: `WgpuToCuda`, the wgpu -> CUDA bridge (feature `cuda-wgpu`), plus RGBA input for `NvEnc` (feature `nvenc`): the zero-copy encode path for a GPU render, the reverse of M220's `CudaToWgpu` and the moat the M267 Bevy demo pays for today (it reads the rendered texture back to system memory and encodes with the ffmpeg NVENC backend). A renderer writes a packed-RGBA `wgpu::Texture` backed by FD-exportable Vulkan memory (`export_rgba_image` / `wrap_rgba_as_texture`, the `R8G8B8A8` mirror of the NV12 export); CUDA imports the same memory as a 4-channel array (`import_rgba_into_cuda`) and `WgpuToCuda::to_cuda_frame` copies it device->device into a fresh linear `CUdeviceptr`, handed downstream as a `MemoryDomain::Cuda` `Rgba8` frame, so the pixels reach the encoder with **no PCIe download**. The bridge retains its own CUDA primary context (the GPU the interop device also selects, released last on drop after the import and the Vulkan image) and owns the exportable render-target texture (`texture()`); each output frame's keep-alive frees its linear buffer on release. `NvEnc` gains RGBA/BGRA input alongside NV12: it negotiates `RawVideo{Rgba8|Bgra8}` (NVENC color converts internally, so still H.264 out) and registers the single packed plane as `ABGR` (wgpu `Rgba8` byte order) / `ARGB`. Deferred: rewiring the `examples/bevy-g2g-stream` demo to render on the interop device (`RenderCreation::Manual`) into the bridge texture (the demo-side plumbing; the standalone test is the validation of record), and registering the bridge as a pipeline element with `MemoryDomain::WgpuTexture` input + a reuse pool (one linear buffer is alloc/freed per frame today).

- M270: `NvDec`, a native NVDEC H.264 decode element (feature `nvdec`, `cfg(linux)`): the decode half of the gst-`nvcodec`-style pair and the mirror of M269's `NvEnc`. It promotes NVIDIA hardware decode from a `FfmpegVideoDec` backend flag (`Backend::NvdecCuda`, which reaches NVDEC *through* libavcodec's cuvid hwaccel) to a first-class element driving the NVIDIA Video Codec SDK's NVCUVID parser+decoder API directly, so the decode path no longer needs libavcodec. With `NvDec -> ... -> NvEnc` both native, the whole H.264 transcode loop stays on the GPU and out of libav. `Caps::CompressedVideo{H264}` (Annex-B) in, `Caps::RawVideo{Nv12}` out in CUDA device memory (`MemoryDomain::Cuda`), the zero-copy hwframe domain `CudaToWgpu` / `CudaGlSink` / `NvEnc` consume with no PCIe download. NVCUVID is callback-driven: a parser (`cuvidCreateVideoParser`) is fed the elementary stream and synchronously invokes a sequence callback (creates the decoder once the SPS geometry is known), a decode callback (`cuvidDecodePicture`), and a display callback (a frame is ready in display order). Because the display callback cannot `await`, it maps the surface (`cuvidMapVideoFrame64`) and pushes a ready frame onto a queue that `process` drains and emits after the parse call returns; the callbacks reach element state through a `*mut DecoderState` passed as the parser's user-data, a pointer that targets a heap `Box` so it stays valid as the runner moves the element between worker threads. Bindings are hand-rolled FFI linking `libnvcuvid` + `libcuda` directly (no `cudarc`, like `nvenc` / the `cuda` module); NVCUVID exports real symbols (no `CreateInstance` table), so the calls are plain `extern "C"`, and the structs are transcribed `#[repr(C)]` with **compile-time size assertions** checked against the installed `cuviddec.h` / `nvcuvid.h` (field offsets verified with `offsetof`), with the per-picture `CUVIDPICPARAMS` left opaque (the parser fills it; we pass the pointer straight to the decoder). Each output frame carries a `CudaKeepAlive` that `cuvidUnmapVideoFrame64`s on drop plus an `Arc` to the decoder, so the decoder / context outlive any frame still in flight (the decoder, its context lock, and the CUDA context are torn down, in that order, only once the last frame is released); the element owns its own CUDA context. The NV12 nanosecond PTS round-trips through the packet/display timestamp; the runtime `CapsChanged` carries the actual cropped display geometry. New on-hardware test on the RTX 3060 runs the **full native loop**: a synthesized CUDA NV12 surface is encoded by `NvEnc` to Annex-B, then decoded by `NvDec` back to CUDA NV12, asserting frame count, the announced NV12 geometry, and (via a small device->host copy) that the decoded luma holds real, non-uniform content; it skips cleanly with no NVIDIA GPU. Pure-logic tests (caps H264->NV12, decoded-vs-negotiated output dims) cover negotiation without a GPU. Like `nvenc` / `ffmpeg` / `cuda`, the `nvdec` feature is CI-excluded. Deferred: mid-stream resolution change (decoder reconfigure), HEVC / other codecs, 10-bit output, and a configurable display-delay (fixed at a low-latency 1 today). This completes the native gst-`nvcodec`-style decode/encode pair.

- M269: `NvEnc`, a native NVENC H.264 encode element (feature `nvenc`, `cfg(linux)`): the zero-copy, device-resident counterpart of `FfmpegH264Enc` and the moat follow-up to M267 (which captured Bevy's render via `from_wgpu` and encoded with `FfmpegH264Enc(NVENC)` *after* a device->host read-back). Where the ffmpeg encoder takes system-memory I420 and copies it into libavcodec, `NvEnc` ingests an NVDEC/CUDA NV12 surface (`MemoryDomain::Cuda`) **in place** and drives the NVIDIA Video Codec SDK (`nvEncodeAPI`) directly, so the pixels never leave the GPU, closing the native `NvdecCuda -> NvEnc` loop with no PCIe download (the encode-side mirror of the M220 `CudaToWgpu` import bridge). `Caps::RawVideo{Nv12}` in, `Caps::CompressedVideo{H264}` Annex-B out; caps do not encode the memory domain, so negotiation matches a system encoder and the frame must be `MemoryDomain::Cuda` at runtime (an `UnsupportedDomain` error otherwise, the symmetric contract `FfmpegH264Enc` upholds for `System`). Bindings are hand-rolled FFI linking `libnvidia-encode` + `libcuda` directly (no `cudarc`, matching the `cuda` module decision): the SDK's giant version-tagged structs are transcribed `#[repr(C)]` with **compile-time size assertions** checked against the installed `nvEncodeAPI.h` (SDK 13.0, verified field offsets via `offsetof`), and the one field-heavy codec-config union is left opaque (the driver fills it via `nvEncGetEncodePresetConfigEx`; we only tweak rate control / GOP). Low latency like `FfmpegH264Enc`: preset P4 + the LOW_LATENCY tuning info, CBR, no B-frames (`frameIntervalP=1`), and an *infinite GOP* so IDRs are emitted only on demand (the first frame + a downstream PLI via `Reconfigure::ForceKeyframe`), each forced IDR carrying in-band SPS/PPS (`OUTPUT_SPSPPS`); the NV12 nanosecond PTS round-trips through NVENC's `inputTimeStamp`. New on-hardware round-trip test on the RTX 3060 synthesizes a CUDA-resident NV12 surface (CUDA driver alloc + upload), encodes it through `NvEnc`, and decodes the Annex-B back through `FfmpegVideoDec` to 320x240 I420; it skips cleanly with no NVIDIA GPU / SDK. Pure-logic tests (caps NV12->H264, non-NV12 / System-domain rejection, the `NVENCAPI_STRUCT_VERSION` macro vs the probed `*_VER` values, the H.264 codec GUID bytes) cover the negotiation and FFI constants without a GPU. Like `ffmpeg`/`cuda`, the `nvenc` feature is CI-excluded (no NVENC runtime in CI). Deferred: system-memory NV12 input (host upload; I420 is already `FfmpegH264Enc`'s job), HEVC via the HEVC GUID, finite-GOP periodic IDRs with `repeatSPSPPS`, an output-bitstream-buffer pool (one is allocated/freed per frame today), and runtime bitrate retarget via `nvEncReconfigureEncoder`. - M265: GPU layer zoo, so a small CNN body runs entirely on-device, not just a single conv. `WgpuInference` could do `linear` (M216) and `conv2d` (M261), but a CNN also needs a nonlinearity and a downsampler, and they had to run without leaving the GPU between layers. - M261: `WgpuInference` gains a GPU-resident 2D convolution, the keystone op for running an actual CNN layer on the device (it could previously only do the `input . W + b` matmul). - M260: `StaticLendRing`, a no-alloc zero-copy DMA capture ring, and an end-to-end heap-free capture path. The strict-`no_std` story had `StaticBufferPool` (move-out, RAII return) and the embassy stack channel (M45), but nothing proved a frame could be *captured and consumed* without the heap. `StaticLendRing::<N, BYTES>` (pure `core`) holds `N` inline slots; the producer fills the next free slot and `publish`es it as a `SystemSlice` that borrows the slot, and a per-slot lease (`AtomicBool`, plain load/store, no CAS, so it builds on `thumbv6m`) is cleared when the lent frame drops, so a slot is reused only after the consumer is done (genuine ring back-pressure: the producer stalls when all slots are in flight). The borrow is runtime-guarded, not a Rust lifetime: `OutputSink::push` and the stack channel take `PipelinePacket` by value (`'static`), so the lend reuses the existing `'static` foreign-buffer carrier (`SystemSlice::from_foreign`) with the lease standing in for the borrow, keeping `Frame` / `MemoryDomain` lifetime-free (no element-signature churn) and requiring zero core-type changes. Unit tests cover lease/recycle and a zero-copy witness (`ring.contains(ptr)`); new `m260_dma_ring` integration test drives a DMA-ring capture `SourceLoop` -> embassy stack channel -> consumer under `block_on`, asserting all frames cross zero-copy (bytes alias the ring), 10 frames recycle <= 4 physical buffers (no per-frame allocation), and every slot is reclaimed after drain. - M258: mid-stream backward coupling through a `DerivedOutput` (the snapshot half of M257). - M257: backward caps coupling through a `DerivedOutput` (decoder / rescaler). - M256: `PoolStage`, a real resizable-pool consumer of the mid-stream β allocation re-cascade.
