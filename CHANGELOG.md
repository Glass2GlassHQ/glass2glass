# Changelog

Pre-release. Work is tracked by milestone (Mn) following the roadmap in `DESIGN.md` §4.10.
Nothing is published yet; all versions are `0.1.0`.

## Unreleased

- M389: `playbin uri=*.ts` multi-stream fan-out + multi-hook playbin dispatch. The `Registry` playbin hook slot becomes a list (`register_playbin` appends, `playbin_hooks()` returns the slice; `parse_launch` tries each in turn until one handles the URI, each declining a container it does not parse), so one hook per container type coexists. New `ts_playbin` hook (`g2g-plugins`): probes a `file://` transport stream's PMT and assembles `FileSrc -> TsDemuxN -> {decode -> auto sink}`, the MPEG-TS sibling of `mkv_playbin`. The two hooks share factored helpers (`open_file_prefix`, `playbin_ports`, `map_playbin_err`); new `TsStreamInfo` / `forwardable_streams` in `tsdemux` (the PMT analog of `mkvdemux`'s). Both registered in `default_registry` (MKV then TS). Tests `m389_playbin_ts`: a lone `playbin uri=file://x.ts` fans out to a 6-node graph, and on one registry a TS file is handled by `ts_playbin` while an MKV file is handled by `mkv_playbin` (each declines the other's container). HLS rides on this: `HlsSrc` emits `ByteStream{MpegTs}`, so `HlsSrc -> TsDemuxN` fans out an HLS variant's streams.
- M388: multi-output MPEG-TS demuxer `TsDemuxN` (`g2g-plugins`), the MPEG-TS sibling of `MkvDemuxN` and the multi-output counterpart of `TsDemux`. A `MultiOutputElement`: one `ByteStream{MpegTs}` in, N elementary streams out, one selected `TsStream` per output port. It parses the transport stream once (reusing the `TsDemuxer` parser + packet realignment) and routes each PES access unit to the port whose codec matches the unit's PMT `stream_type`, emitting each port's elementary `Caps` (`CapsChanged`, via `port_output_caps` for startup branch negotiation) before its first frame; a stream no port carries is dropped, an absent stream's port stays dark. With a bus, announces the same M386 `StreamCollection`. So one TS demuxer can feed audio + video decode branches in one pipeline (the `playbin uri=*.ts` fan-out core, M389). Tests `m388_tsdemuxn`: a hand-built A/V transport stream (PAT + 2-stream PMT + video/audio PES) splits onto two ports (H.264 AUs on port 0, AAC on port 1, each with one CapsChanged, collection announced once), and a video-only multiplex leaves the audio port dark.
- M387: gapless playbin convenience builder `gapless_playbin` (`g2g-plugins`), the one-call front door to the M383-M386 gapless stack (which was hand-wired before). Given a registry, a playlist of URIs, a sink, and a target shape, it builds the first URI's source, wraps it in a `GaplessSrc`, enqueues the remaining URIs' sources on a shared `GaplessController`, and auto-plugs one decode chain (reused across items) to the sink, returning the runnable `GaplessSrc -> decode -> sink` graph plus the controller (so the app can `enqueue` more, `switch_now`, and must `finish()` when done). Factors a core `Registry::build_source_decodebin` out of `build_uridecodebin` (auto-plug `decode -> sink` from an already-built source) for the wrapped-source case. New `GaplessPlaybinError` (empty playlist / URI). Tests `m387_gapless_playbin`: a 3-URI playlist builds a 3-node graph with the two successors pre-enqueued, and empty-playlist / unknown-scheme / bad-later-item are rejected. Completes the multi-stream playback arc (M376-M387).
- M386: stream discovery for MPEG-TS and MP4, the M376 `StreamCollection` pattern extended past Matroska. `TsDemux` gains `with_bus` and announces every elementary stream the PMT declares as a `BusMessage::StreamCollection` (id `mpegts-pid-N`, kind + media-type caps mapped from the PMT `stream_type`, geometry `Any` until refined), once when the PMT parses. `Mp4Src` (already bus-attached for tags) announces its single video track (`mp4-track-0`, the probed codec + geometry) alongside its tag post. So an app can discover the streams of a `.ts` / `.mp4` the same way it does an MKV (the discovery half of multi-stream playback; a multi-output TS demuxer + a `playbin` TS/MP4 fan-out hook remain follow-ups). Tests `m386_stream_collection`: a PAT + 2-stream PMT (H.264 + AAC) yields one collection listing both streams keyed by PID, and an Mp4Mux-written H.264 file read back through `Mp4Src` yields a one-stream video collection.
- M385: A/V offset transform `AvOffset` (`g2g-plugins`), the g2g form of GStreamer playbin's `av-offset`. A 1-in/1-out pass-through (wildcard `IdentityAny` caps, like `identity`) that shifts every `DataFrame`'s PTS/DTS by a signed `offset` ns: positive delays the stream, negative advances it (saturating, clamped at 0 since a PTS cannot go negative), `0` is a no-op pass-through. Put one on a branch of a multi-stream playback graph (typically audio) to re-align it against the others. Registered as `avoffset` with a settable `offset` (`PropKind::Int`). Tests `m385_avoffset`: positive offset delays PTS+DTS (`[0,1000,2000]`->`[500,1500,2500]`), negative advances + clamps (`[0,1000]` at `-500`->`[0,500]`), zero is pass-through, and the element is registered with a round-tripping `offset` property + builds from `audiotestsrc ! avoffset offset=40000000 ! fakesink`.
- M384: instant (flushing) URI switch on `GaplessSrc`, the `instant-uri` analog (vs the M383 gapless `enqueue`). New `GaplessController::switch_now(source)` (a latest-wins instant slot) + `wait_instant` future; `GaplessSrc::run` now races the current item's `run` against `wait_instant` via `select2`, so a `switch_now` preempts the playing item: `select2` drops the losing `run` future, cancelling the inner source mid-stream (its remaining frames are abandoned, the instant-uri contract), then the source pushes a `Flush`, resets the timeline to 0, and plays the requested source. A `switch_now` between items also flushes + resets (vs `enqueue`'s gapless continuation); `wait_event` now also wakes on `switch_now`, and about-to-finish is suppressed when an instant switch is pending. Frame counting moved into the `ShiftSink` adapter (the inner's own count is lost when a preemption drops its future). Tests: `gapless` controller units (`switch_now` latest-wins + take-once, `wait_instant` resolves only on `switch_now` not `enqueue`) and `m384_instant_switch` (a long clip1 is preempted mid-playback by `switch_now(clip2)` — a concurrent app polls a shared frame counter and switches after 2 frames — asserting exactly one flush, clip1 truncated well before its 100 frames, and clip2 playing from a reset timeline `[0,500,1000]` then a single terminal Eos).
- M383: gapless playback, the playbin `about-to-finish` + next-`uri` analog. New `g2g-core` `GaplessController` (a cloneable app<->source channel mirroring `SeekController`: a `VecDeque` playlist the app `enqueue`s, an about-to-finish back-channel `notify`/`take`, a latching `finish` flag, and a wakeful `wait_event` so an idle source parks rather than busy-polls) and a `GaplessSrc` element (`g2g-plugins`) that wraps a current `DynSourceLoop` + a `GaplessController`. It plays the current item, posts about-to-finish when nothing is queued behind it (so the app can enqueue the next item *during* playback, a seamless swap), and on the item's EOS pulls the next, rebasing that item's PTS/DTS onto the running timeline (the previous item's end becomes the new item's zero) via an interposing `ShiftSink` that also swallows the inner item's `Eos`; the only terminal `Eos` is emitted once the playlist is `finish`ed and the queue drains, so the downstream decode chain is reused across items with no flush and no gap (the read-side analog of GStreamer reusing one decodebin across URIs). Builds directly on the M211/M358 running-time machinery (this concatenates *different* items where the segment loop loops *one*). Scope v1: items share a codec (decode chain reused; a per-item caps refinement still flows via the inner source's `CapsChanged`); instant (flush) URI switching and an A/V offset are follow-ups. Tests: `gapless` controller unit tests (FIFO enqueue/take, about-to-finish take-once, finish latches, `wait_event` wakes on enqueue/finish) and `m383_gapless` (a pre-loaded two-clip playlist concatenates to rebased PTS `[0,1000,2000,3000,4000]` with a single Eos and no about-to-finish needed, and a dynamic playlist where the app enqueues each successor in response to about-to-finish then finishes, concatenating gaplessly across the swap).
- M382: `playbin uri=X` auto-fan-out, the gst-launch DSL front door to the M376-M381 multi-stream stack. A lone `playbin uri=file://x.mkv` in `parse_launch` probes the container and auto-builds `FileSrc -> MkvDemuxN -> {decode -> auto sink}` with one decode branch per forwardable stream, the multi-stream form of `playbin` (vs the M196 single-stream expansion). Cross-crate by design (the text DSL is in `g2g-core`, Matroska parsing in `g2g-plugins`): core gains a `Registry` `PlaybinHook` fn-pointer slot (`register_playbin` / `playbin_hook`, `Default`-friendly), and `parse_launch` calls it for a lone `playbin` (a `playbin` mid-pipeline or without `uri=` is left to the single-stream builder; `Ok(None)` / no hook falls back to single-stream `playbin`). `build_playbin_graph` is refactored to expose `build_playbin_graph_with_source`, so the hook supplies a Matroska byte `FileSrc` directly instead of the `file://` handler's MP4-self-demuxing source. The plugin hook `mkv_playbin` reads a bounded 4 MiB prefix, parses `Tracks`, and (new `forwardable_streams` / `MkvStreamInfo` in `mkvdemux`) builds one `PlaybinPort` per track (autovideosink for video, autoaudiosink for audio); a non-`file://` URI, unreadable file, or non-Matroska container declines. Tests `m382_playbin_uri`: a real two-track (H.264 + AAC) MKV fans out to a 6-node graph, a registry without the hook (or a non-Matroska file) stays single-stream (3 nodes), and a non-lone `playbin` never fires the hook. Gapless URI switching is the next follow-up (M383).
- M381: app-driven stream selection on the multi-output demuxer `MkvDemuxN::with_stream_select`, the multi-output counterpart of M377 (single-output `MkvDemux` switch). A `StreamSelectController` re-maps which stream each port carries: port `i` takes the `i`-th selected stream id (from the announced M376 collection), the demuxer re-arms that port's `CapsChanged` when its stream changes, and confirms the active ids on the bus (`BusMessage::StreamsSelected`). The id->`MkvStream` resolution and codec mapping are factored into shared `resolve_stream_id` / `codec_to_stream` free functions used by both demuxers. Switching a port to a different *codec* needs its downstream decode branch re-plugged (the decodebin re-plug, a follow-up); a same-codec switch takes effect directly. Test `m381_mkvdemuxn_select`: a two-port demux starts `[video, audio]`, the app swaps it to `[audio, video]` after the collection is announced, and the Clusters re-route so port 0 carries AAC and port 1 H.264, with `StreamsSelected` confirming the per-port ids. The remaining playbin gap: a `playbin` gst-launch DSL node.
- M380: per-branch decoder negotiation, so a real decoder configures against its codec downstream of a demux node (the gap M379 left). A multi-output demux now declares each port's elementary-stream caps via `MultiOutputElement::port_output_caps` (default `None` = a broadcast fan-out like `Router`); `MkvDemuxN` returns each port's `MkvStream` caps. The graph solver gains a `NodeConstraint::Demux { input, ports }` (built when a demux declares every port): instead of a broadcast tee coupling `in == every out`, it narrows each branch's out-edge to its port's `Produces` caps decoupled from the byte-stream input (`apply_demux_node`, the `node_consistent` / `graph_downstream_feasibility` tee arms made demux-aware). The startup configure pass then hands each branch element its port caps, so a decoder that *requires* H.264 / AAC (and rejects the byte stream) configures at startup rather than having to be accept-anything and retype only at runtime. A pass-through fan-out (declaring no port caps) stays a plain tee. Test `m380_branch_negotiation`: `BytesSrc(Matroska) -> MkvDemuxN(H.264, AAC) -> picky decoder -> sink` per port, where each `PickyDecoder` rejects anything but its codec, so the graph runs only because per-branch negotiation handed it the port caps (asserted: each decoder configured with its elementary caps, frames flow). The remaining playbin gaps: wiring `StreamSelectController` into `MkvDemuxN` and a `playbin` gst-launch DSL node.
- M379: `playbin`-equivalent graph builder `Registry::build_playbin_graph`, the multi-stream counterpart of `build_uridecodebin`. Given a URI, a multi-output demuxer (a `MultiOutputElement`, e.g. `MkvDemuxN`), and one `PlaybinPort` per selected stream (`{ input_caps, target, sink }`, derived by the app from the demux's announced M376 `StreamCollection` + its M377 selection), it assembles `source -> demux -> {auto-plugged decode chain -> sink}` per port and returns a `Graph` for `run_graph`. Reuses the existing `build_uri_source` (URI dispatch), `Graph::add_demux` (the M210 demux node), `Registry::decodebin` (per-port chain splice), and `run_graph` (the M210 demux arm broadcasts EOS, each branch retypes from the byte-stream input via the demux's per-port `CapsChanged`). New `PlaybinPort` / `PlaybinGraphError` types, exported from `g2g_core::runtime`. Per-branch *static* negotiation against the port caps (so a real codec decoder, not just an accept-any branch, configures at startup rather than retyping at runtime) is the remaining gap, the M10 per-branch-negotiation follow-up; wiring a `playbin` node into the `gst-launch` DSL also remains. Tests `m379_playbin`: the builder assembles the right topology (source + demux(2) + two stub decoders + two sinks, 5 edges) and rejects no-ports / unknown-scheme, and `MkvDemuxN` runs as a first-class `run_graph` demux node end-to-end (a `BytesSrc` feeds a muxed H.264 + AAC stream through the demux to two counting sinks, two access units on each port).
- M378: multi-output Matroska demuxer `MkvDemuxN` (the decodebin core, the read-side analog of `MkvMuxN`). A `MultiOutputElement` driven by `run_source_fanout`: one Matroska / WebM byte stream in, N elementary streams out, one selected `MkvStream` per output port (the §4.9.3 "dark slots"). It parses the container once and routes each track's access units to its port by codec, so one demuxer feeds several decode branches in one pipeline (audio + video together) instead of one single-output demuxer per stream. Each port emits its concrete `Caps` (`CapsChanged`) before its first frame (the branch retypes from the byte-stream input caps to the elementary stream); a parsed track no port carries is dropped, and a port whose stream the container lacks stays dark. Reuses `MkvDemux`'s codec/caps mappings (`selected_codec`, `output_caps`, `track_to_stream`, and a `concrete_caps` refactored into a shared `concrete_caps_of(demux, stream)`); with a bus, announces the same M376 `StreamCollection`. Per-branch caps negotiation (each branch sink still configures with the byte-stream caps, retyping at runtime) and wiring a `demux`/`playbin` node into the `gst-launch` DSL are the follow-ups (M379). Tests: `m378_mkvdemuxn` mux an H.264 + AAC stream with `MkvMuxN`, demux through a two-port `MkvDemuxN` (port 0 = H.264, port 1 = AAC), asserting each port gets its own CapsChanged and only its track's frames, the collection is still announced, and an unselected/absent stream leaves its port dark.
- M377: app-driven stream selection, the playbin `SELECT_STREAMS` analog (building on M376 discovery). New `g2g-core` `StreamSelectController` (a cloneable app->demuxer channel mirroring `SeekController`: `select(ids: Vec<String>)` / `take_pending`, latest-wins, poll-based) and a `BusMessage::StreamsSelected { ids }` confirmation (the `GST_MESSAGE_STREAMS_SELECTED` analog). `MkvDemux::with_stream_select` makes its single output app-selectable: it polls the controller in `emit_ready`, resolves a published id (`matroska-track-N`) to the track's `MkvStream`, switches which stream it forwards (clearing `last_caps` so the new stream's caps re-emit as a `CapsChanged`), and confirms the active id on the bus. Switching to a different-kind stream re-negotiates caps (video->audio) at the demuxer; a real single decoder can only follow a same-kind switch, the cross-kind case is what the M378 multi-output demuxer is for. `TsDemux`/`Mp4Src` don't yet surface their full stream lists (TsDemux filters by `stream_type` at emit, Mp4Src is single-track progressive), so announcing + selecting their streams is a follow-up. Tests: `stream_select` unit tests (latest-wins take, clone shares the slot) and `m377_stream_select` (mux an H.264 + AAC stream, drive `MkvDemux` header-first so it announces the collection with no frames, then select the audio id and feed the Clusters: the output switches mid-stream, emitting a video->audio CapsChanged, forwarding the AAC payloads not the video, and posting `StreamsSelected`).
- M376: stream discovery, the multi-stream playback foundation (the stream-collection model that extends the single-stream playbin g2g already has, M196). New `g2g-core` baseline types (`StreamType` {Video, Audio, Text, Unknown}, `Stream` {id, stream_type, caps}, `StreamCollection` {id, streams} with `streams_of_type` / `get` / `len`), the `GstStreamCollection` analog, and a `BusMessage::StreamCollection` (the `GST_MESSAGE_STREAM_COLLECTION` analog; `BusMessage` drops its `Eq` derive since `Caps` is only `PartialEq`, nothing keyed on it). `MkvDemux` is the reference: it announces *every* track the container declares (id `matroska-track-N`, kind + caps from the codec) on the bus once `Tracks` is parsed, regardless of which single stream it forwards, mirroring its existing `post_tags` (kept once across a mid-segment seek's `reset_keeping_tracks`). App-driven selection among the streams, multi-output routing, the `playbin` element, and gapless are follow-ups (M377+); `TsDemux` (PMT) / `Mp4Src` (moov) announce via the same pattern. Tests: `stream` unit tests (filter-by-type, get-by-id, empty) and `m376_stream_collection` (mux an H.264 + AAC stream with `MkvMuxN`, demux it with a bus, assert one collection listing both streams with the right kind / caps, announced exactly once).
- M375: the Matroska muxer writes a `Cues` index, closing the seek round-trip (the write side of M373's read). `MatroskaMuxer` tracks each Cluster's byte offset (relative to the Segment data start, the anchor `cue_seek_offset` adds back) as it streams and records a `CuePoint` for the first keyframe in each Cluster on the cue track (the first video track, else track 0; deduped per Cluster to bound the index). A new `finish()` returns the `Cues` element, which `MkvMux` / `MkvMuxN` flush as one final frame at EOS (after the last Cluster). A streaming, unknown-size-Segment muxer cannot place a front `SeekHead` pointing at the end-of-stream `Cues` (the front is already emitted before the `Cues` position is known), so seeking the muxer's own output uses the M373 read-to-end-then-seek path, not the M374 `SeekHead` prefetch; a front `SeekHead` (seekable from byte 0 without reading past the Clusters) would need a two-pass / seekable-output finalize mode (open). Tests: a `matroska` unit test muxing two Clusters then asserting the written `Cues` resolve (via `cue_seek_offset`) to the exact Cluster element ids, and `m375_mkvmux_cues` driving the full round-trip (`MkvMux` -> bytes -> `MkvDemux` with `with_seek`: a seek lands on Cluster 2 through the muxer-written index, not a re-scan from 0).
- M374: Matroska `SeekHead`-driven `Cues` prefetch, making the M373 index usable on the first seek of a real (end-of-`Cues`) file. The `matroska` parser reads a `SeekHead` (a `Seek` entry whose `SeekID` is `Cues`) into the `Cues` byte position, exposing `cue_index_offset()` (absolute). `MkvDemux` services a seek in three tiers (`poll_seek`): `Cues` already parsed -> seek straight to the target Cluster (M373); only a `SeekHead` locating `Cues` -> a two-hop prefetch (byte-seek to `Cues`, parse, then byte-seek to the target Cluster), the internal prefetch flush consumed rather than forwarded so downstream sees a flush only on the real seek; neither -> re-scan from 0 (M364). `DemuxSeek` gains `is_seeking` / `begin_indexed_seek` for the demuxer-driven second hop; `MkvDemux` holds its own controller clones for the prefetch byte-seeks. Tests: `matroska` unit tests for `SeekHead` -> Cues position, and `m374_mkvdemux_seekhead_prefetch` driving the full SeekHead -> Cues -> target two-hop (asserting both byte-seek offsets, the consumed-vs-forwarded flushes, and the re-sync). (The muxer writing `Cues` / `SeekHead` so its own output is seekable remains a follow-up.)
- M373: Matroska `Cues`-indexed seeking. The `matroska` parser now reads the `Cues` element into a `Vec<CuePoint>` (time -> Cluster byte position, the position kept relative to the Segment data start) and tracks the Segment's absolute data offset, exposing `MatroskaDemuxer::cue_seek_offset(target_ns)` (the absolute byte offset of the Cluster with the largest `CueTime` at/before the target). `MkvDemux` seeks through it: `DemuxSeek` gains `poll_request_indexed`, so when the index is known the demuxer byte-seeks straight to the target Cluster (mid-segment) and `reset_keeping_tracks` preserves Tracks / TimestampScale / Cues across the landing, instead of the M364 re-scan from offset 0 (the fallback when no `Cues` are parsed, via `poll_request`/`keeps_state`). Tests: `matroska` unit tests for `Cues` parse + `cue_seek_offset` (SNAP-before, clamp, no-index -> None), and `m373_mkvdemux_cues_seek` driving the demuxer through a full-file feed (parsing `Cues`) then a seek that requests the *Cluster1* offset (not 0) and re-syncs there. End-of-file `Cues` need read-past or a `SeekHead` prefetch (a follow-up); the muxer does not yet write `Cues`.
- M372: throughput-driven ABR for `DashSrc` (`with_abr()`), reusing the shared `abr::BandwidthEstimator` from M371. A `load_rep` helper resolves any Representation (Template / List / `sidx`-fetched SegmentBase) into the run loop's `(segments, timescale, init)` working set, so ABR is "re-select the Representation whose declared bandwidth fits the estimate, re-resolve it, re-emit the init, keep the time-aligned index". The estimate-derived cap drives both the per-reload pick and a per-segment re-selection (so a static VOD also adapts within one pass). Off by default. Test: an end-to-end `m160` test over a bandwidth-throttled two-Representation MPD that starts on the high rendition then downswitches to the low one (re-emitting its init) after the slow segment.
- M371: throughput-driven ABR for `HlsSrc` (`with_abr()`), with a shared `abr::BandwidthEstimator` for the adaptive sources. The estimator keeps an EWMA of measured download throughput (bytes over elapsed `monotonic_ns`, the caller supplies the timing so it stays pure `no_std`) and turns it into an effective bandwidth cap (estimate scaled by a safety factor, bounded by the user's `max-bandwidth`). ABR reduces to feeding that cap to the *existing* `MasterPlaylist::select`: the run loop times each segment download, re-selects the best-fitting variant, and on a change swaps the active media playlist and re-emits the init segment, keeping the time-aligned segment index. Off by default (a fixed up-front variant, unchanged behaviour); a media-only playlist has nothing to switch. Tests: estimator unit tests (EWMA, safety cap, user-cap bound, zero-sample) and an end-to-end `m156` test over a bandwidth-throttled two-variant master that starts on the high variant then downswitches to the low one after measuring the slow download. (DASH ABR reuses the estimator in a follow-up.)
- M370: DASH `SegmentBase` (`sidx`-indexed single-file CMAF), completing DASH byte-range addressing. A `SegmentBase` Representation is one resource whose media-fragment byte ranges are not in the manifest but in a `sidx` (Segment Index) box at `indexRange`. New `SegmentSource::Base` carries the `indexRange` + `<Initialization>` range; at run time `DashSrc` fetches the index range, parses the `sidx` (new `parse_sidx`, bounds-checked against untrusted input per the parser policy: a malformed box / hostile `reference_count` fails to `None`), and `Sidx::subsegments` turns the referenced sizes + durations into byte ranges of the `BaseURL` (anchored at `indexRange + box_size + first_offset`) with cumulative times in the `sidx` timescale. These feed the same `ResolvedSegment` path as Template / List, so the run loop, byte-range fetch, and M367 seek work unchanged; the index bytes themselves are never pushed downstream. Tests: `mpd` unit tests for `sidx` parse + subsegment resolution, truncated/wrong-box rejection, and `SegmentBase` parsing; an end-to-end `m160` test building an `[init][sidx][frags]` resource served with `206` Range that reassembles init+frags (skipping the index) and demuxes to the access units.
- M369: DASH `SegmentList` byte-range addressing (the DASH counterpart of M368). The `mpd` parser generalizes `Representation` from a single `SegmentTemplate` to a `SegmentSource` enum (`Template` | `List`); `SegmentList` is an explicit ordered list of `<SegmentURL>` entries, each a `@media` URL and/or a `mediaRange` byte range of the `BaseURL` resource, with an `<Initialization>` (`sourceURL` / `range`). New `Representation` accessors (`timescale`, `init`, `resolved_segments`, `template`) drive both addressing modes through one `ResolvedSegment { url, byte_range, time }` path, so the `DashSrc` run loop and the M367 seek work unchanged across both. A range-only entry (empty `@media`) fetches just its sub-range of the BaseURL via `fetch::get_range_bytes` (now shared by `hls` + `dash`); inclusive `"start-end"` ranges parse via `parse_dash_range`, rejecting reversed/malformed ones. Tests: `mpd` unit tests for SegmentList parsing (byte-range and explicit-URL forms) + range parsing, and an end-to-end `m160` test serving one resource with `206` Range support that reassembles byte-exact and demuxes to the access units. (`SegmentBase`'s `sidx`-indexed single resource stays open.)
- M368: HLS `#EXT-X-BYTERANGE` single-file CMAF. A media segment (or the `#EXT-X-MAP` init segment via its `BYTERANGE` attribute) that addresses a byte sub-range of a resource now fetches only that range with an HTTP `Range` request, so a single-file CMAF stream (init + all fragments packed in one resource) plays. The `hls` parser gains a `ByteRange { offset, length }` on `Segment` / `MediaPlaylist`; an absent `@offset` continues from the previous sub-range of the same URI (RFC 8216 §4.3.2.2). New `fetch::get_range_bytes` issues the `Range` request, caps the body like `get_bytes`, and is robust to a server that ignores the range (replies `200` with the whole resource) by slicing the requested window. Tests: a parser unit test for implicit/explicit offsets, and an end-to-end `m156` test serving one resource with `206 Partial Content` Range support that reassembles to the byte-exact stream and demuxes back to the access units. (DASH `SegmentList` / `SegmentBase` byte-range is the still-open counterpart.)
- M367: adaptive sources `HlsSrc` / `DashSrc` are time-seekable (`with_seek`). Unlike the BYTES-format `FileSrc`, an app TIME seek resolves to the media segment containing the target: HLS walks cumulative `#EXTINF` durations, DASH maps the target onto the `SegmentRef` `$Time$` line. The source then emits `Flush`, jumps to that segment, re-emits the fMP4 `#EXT-X-MAP` / init segment (a downstream `fmp4demux` reset on the flush needs its `moov` again), emits the post-flush `Segment` at the segment start, and resumes there. Clamped to the last segment (a target past the end lands there). The CMAF / DASH segment-transition case; discontinuity / multi-period `SEGMENT` emission stays separate. - M351: two-sided allocation-domain negotiation. The memory domain of an allocation was a one-sided dictate: each consumer named exactly one domain, the producer silently obeyed (`configure_allocation` can't refuse), and a tee whose branches named different domains failed loud (`AllocationConflict`) even when a domain satisfying both existed. Domain choice is now a negotiation. All defaults reduce to today's exact-match behaviour, so a single-domain pipeline negotiates the identical domain it did before. - M350: backtracking fixation for true diamonds (DAG solver completeness). `solve_graph` ran arc consistency (pairwise per-edge narrowing) and then fixated each edge *independently* with `CapsSet::fixate`. The fixation step is now a bounded **backtracking search** over the arc-consistency-narrowed domains: it assigns one fixated `Caps` per edge in id order, tries each edge's greedy choice first (so a chain or independent fan-out fixates byte-for-byte as before), and prunes the moment a fully-assigned node violates its cross-edge relation (`node_consistent` / `transform_pair_consistent`: a tee's branches all carry its input; a transform's `(in, out)` is a real `Mapping` pair / `Identity` equality / derived `f(in)` image). - M335: `CapabilityDescriptor` for auto-plug element selection. - M334: FlexFEC (RFC 8627) forward error correction. New sans-IO `flexfec` module, the wide-mask sibling of ULPFEC: a repair packet rides a dedicated FEC SSRC (its own RTP stream) and a variable-length protection bitmask (15 / 46 / 109 bits, the per-word continuation flag picking the shortest form) names which media packets, by sequence offset from an `SN base`, it covers. One repair therefore protects up to 109 packets, past ULPFEC's 16, and arbitrary strided / 2-D patterns are expressible. `build_flexfec_packet` / `recover_packet` XOR the same RTP fields as ULPFEC (P/X/CC, M/PT, timestamp, length, payload); `FlexFecEncoder` emits one repair per group and `FlexFecDecoder` buffers media + repairs and chains recoveries, so 2-D (row + column) repairs reconstruct a multi-loss row. Wired into both elements: `UdpSink::with_flexfec(group, pt, ssrc)` (a `FecMode::FlexFec`) and `UdpSrc::with_flexfec(pt)` (a parallel decoder routed by payload type, alongside the ULPFEC path). Tests: mask round-trips across all three widths, single-loss recovery, a 24-packet group recovered by one repair (impossible for ULPFEC), an encoder/decoder round trip, and 2-D recovery of two losses in one row via column repairs; an end-to-end `flexfec_recovers_a_loss_in_a_group_beyond_ulpfec` loopback recovers a drop in a 20-packet group through a one-way (no-feedback, no-retransmit) path. Real-peer interop unverified (sandbox); validated g2g <-> g2g.
- M333: SRT live-mode congestion control (output pacing). SRT live mode does not use the window-based AIMD of file transfer; it paces packets so the egress does not burst-flood the path and leaves headroom for retransmits. New sans-IO, integer-only `LiveCc`: it either caps at a fixed bandwidth or follows the measured input rate (an EWMA over per-packet byte/time samples) plus an overhead margin, and reports `snd_period_us(len)`, the microseconds to wait before the next packet to hold the target rate (0 before the estimate warms up = send freely). `SrtSink::with_max_bandwidth(bytes_per_sec)` enables it (SRT semantics: a positive value is a hard cap, `0` is rate-follow with 25% headroom); the send loop tracks a `next_send_us` cursor and `tokio::time::sleep`s each packet to its scheduled slot. Tests: unit tests for the fixed cap (1 MB/s -> 1000 B = 1 ms), rate-follow tracking + overhead (1.25 MB/s input, 25% headroom -> 800 us for a 1250 B packet), and no-pacing-before-warmup; an end-to-end `congestion_control_paces_egress_to_the_bandwidth_cap` loopback that sends 30 packets with no inter-frame sleep under a 200 kB/s cap and asserts the caller's wall-clock elapsed is held near the expected ~197 ms (a lower bound robust to scheduler slack), where an unpaced loop would finish ~instantly.
- M332: SRT mid-stream key rotation. The stream cipher key no longer lives for the whole session: `SrtSink::with_key_rotation(every_packets)` rolls to a fresh random key periodically so a recovered key exposes only a bounded window. Rotation rides SRT's two key slots (`KK` even/odd): `SrtSender`/`SrtReceiver` now hold both an even and an odd `SrtCrypto`, each data packet is stamped and decrypted by its `KK` parity, and a rekey installs the new key into the *inactive* slot then switches to it, so packets still in flight (and NAK retransmits) under the previous key keep decrypting until the next rotation overwrites that slot. The sender buffers `(seq, kk, ciphertext)` so a retransmit carries the parity it was originally encrypted under, not the current active key. A new `Control::KeyMaterial { rsp, km }` control packet (SRT user/extended type `0x7FFF`, subtype KMREQ/KMRSP) carries the rekey: `SrtSender::rekey` builds it (via `SrtCrypto::build_km(pass, kk)`) and announces it before the first packet under the new key; `SrtSrc` unwraps it (`SrtCrypto::km_kk` + `from_km`) and installs it into the named slot. Tests: a sans-IO `rekey_switches_keys_and_receiver_decrypts_across_the_switch` (even->odd flip, wire `KK` flips, all payloads decrypt) and a `KeyMaterial` control round-trip; an end-to-end `key_rotation_rolls_the_cipher_mid_stream` loopback rekeying every 3 packets where the listener installs each new key from the KM and decrypts all 12 payloads in order across `>= 2` rotations. Lossless-path rekey only; KM retransmit-until-KMRSP for a lossy rekey announcement is a follow-up.
- M331: AES-256 for the SRT stream cipher. `srtcrypto` was AES-128 only; the SEK is now an `Aes128([u8;16])` / `Aes256([u8;32])` enum and `process` selects AES-128-CTR or AES-256-CTR (the 128-bit CTR counter block is key-size independent, only the key schedule differs). The KM (Keying Material) message already carries `KLen`, so the two sizes interoperate on one wire layout: `build_km` records the key length and the KEK is derived to match (PBKDF2-HMAC-SHA1 to 16 or 32 bytes, wrapped with `Kek::<Aes128>` / `Kek::<Aes256>` RFC 3394), and `from_km` reads `KLen` to rebuild the right cipher, so the listener picks up AES-256 from the handshake with no extra config. `SrtSink::with_aes256()` opts in (default stays AES-128); `SrtCrypto::generate` takes an `AesKeySize`. `build_km` also gained a key-parity (`KK`: even/odd) argument and a `km_kk` reader, the groundwork for mid-stream rekeying. - M330: interleaved (column) ULPFEC for burst-loss recovery. Single-level ULPFEC emits one repair per contiguous group and recovers only one loss per group, so a burst of consecutive losses defeats it. `InterleavedFecEncoder` buffers a `rows x stride` block (capped at 16, the level-0 mask width) and emits `stride` *column* repairs, where column `c` protects the strided set `c, c+stride, c+2*stride, ...`; a burst of up to `stride` consecutive losses then hits at most one packet per column, so each column repair recovers its loss and the whole burst is reconstructed. `build_fec_packet` now derives the protection mask from each packet's actual sequence offset from the SN base (identical bits for a contiguous run, so existing single-level behavior is unchanged) and rejects a set spanning more than 16 sequence numbers; the decoder needed no change (it already reads arbitrary masks and chains recoveries). `UdpSink::with_interleaved_fec(rows, stride, pt, ssrc)` wires it in (the sink's `fec` field became a `FecMode` enum: None / Single / Interleaved). - M272: registry / auto-plug wiring for the native `NvEnc` / `NvDec` (M269 / M270), so `gst-launch` / `gst-inspect` / `decodebin` see them. - M270: `NvDec`, a native NVDEC H.264 decode element (feature `nvdec`, `cfg(linux)`): the decode half of the gst-`nvcodec`-style pair and the mirror of M269's `NvEnc`. It promotes NVIDIA hardware decode from a `FfmpegVideoDec` backend flag (`Backend::NvdecCuda`, which reaches NVDEC *through* libavcodec's cuvid hwaccel) to a first-class element driving the NVIDIA Video Codec SDK's NVCUVID parser+decoder API directly, so the decode path no longer needs libavcodec. With `NvDec -> ... -> NvEnc` both native, the whole H.264 transcode loop stays on the GPU and out of libav. `Caps::CompressedVideo{H264}` (Annex-B) in, `Caps::RawVideo{Nv12}` out in CUDA device memory (`MemoryDomain::Cuda`), the zero-copy hwframe domain `CudaToWgpu` / `CudaGlSink` / `NvEnc` consume with no PCIe download. NVCUVID is callback-driven: a parser (`cuvidCreateVideoParser`) is fed the elementary stream and synchronously invokes a sequence callback (creates the decoder once the SPS geometry is known), a decode callback (`cuvidDecodePicture`), and a display callback (a frame is ready in display order). Because the display callback cannot `await`, it maps the surface (`cuvidMapVideoFrame64`) and pushes a ready frame onto a queue that `process` drains and emits after the parse call returns; the callbacks reach element state through a `*mut DecoderState` passed as the parser's user-data, a pointer that targets a heap `Box` so it stays valid as the runner moves the element between worker threads. Bindings are hand-rolled FFI linking `libnvcuvid` + `libcuda` directly (no `cudarc`, like `nvenc` / the `cuda` module); NVCUVID exports real symbols (no `CreateInstance` table), so the calls are plain `extern "C"`, and the structs are transcribed `#[repr(C)]` with **compile-time size assertions** checked against the installed `cuviddec.h` / `nvcuvid.h` (field offsets verified with `offsetof`), with the per-picture `CUVIDPICPARAMS` left opaque (the parser fills it; we pass the pointer straight to the decoder). Each output frame carries a `CudaKeepAlive` that `cuvidUnmapVideoFrame64`s on drop plus an `Arc` to the decoder, so the decoder / context outlive any frame still in flight (the decoder, its context lock, and the CUDA context are torn down, in that order, only once the last frame is released); the element owns its own CUDA context. The NV12 nanosecond PTS round-trips through the packet/display timestamp; the runtime `CapsChanged` carries the actual cropped display geometry. New on-hardware test on the RTX 3060 runs the **full native loop**: a synthesized CUDA NV12 surface is encoded by `NvEnc` to Annex-B, then decoded by `NvDec` back to CUDA NV12, asserting frame count, the announced NV12 geometry, and (via a small device->host copy) that the decoded luma holds real, non-uniform content; it skips cleanly with no NVIDIA GPU. Pure-logic tests (caps H264->NV12, decoded-vs-negotiated output dims) cover negotiation without a GPU. Like `nvenc` / `ffmpeg` / `cuda`, the `nvdec` feature is CI-excluded. Deferred: mid-stream resolution change (decoder reconfigure), HEVC / other codecs, 10-bit output, and a configurable display-delay (fixed at a low-latency 1 today). This completes the native gst-`nvcodec`-style decode/encode pair.

- M269: `NvEnc`, a native NVENC H.264 encode element (feature `nvenc`, `cfg(linux)`): the zero-copy, device-resident counterpart of `FfmpegH264Enc` and the moat follow-up to M267 (which captured Bevy's render via `from_wgpu` and encoded with `FfmpegH264Enc(NVENC)` *after* a device->host read-back). - M260: `StaticLendRing`, a no-alloc zero-copy DMA capture ring, and an end-to-end heap-free capture path. The strict-`no_std` story had `StaticBufferPool` (move-out, RAII return) and the embassy stack channel (M45), but nothing proved a frame could be *captured and consumed* without the heap. `StaticLendRing::<N, BYTES>` (pure `core`) holds `N` inline slots; the producer fills the next free slot and `publish`es it as a `SystemSlice` that borrows the slot, and a per-slot lease (`AtomicBool`, plain load/store, no CAS, so it builds on `thumbv6m`) is cleared when the lent frame drops, so a slot is reused only after the consumer is done (genuine ring back-pressure: the producer stalls when all slots are in flight). The borrow is runtime-guarded, not a Rust lifetime: `OutputSink::push` and the stack channel take `PipelinePacket` by value (`'static`), so the lend reuses the existing `'static` foreign-buffer carrier (`SystemSlice::from_foreign`) with the lease standing in for the borrow, keeping `Frame` / `MemoryDomain` lifetime-free (no element-signature churn) and requiring zero core-type changes.
