## Unreleased

- M244: WebRTC adaptive bitrate / congestion control (parity track T2). New reverse-channel signal `PushOutcome::Bitrate(u32)` + `BitrateSlot` + `AsyncElement::take_bitrate` (bits/second), polled by the runners alongside `take_qos` / `take_reconfigure` (priority: Reconfigure > Qos > Bitrate). `WebRtcSink` enables str0m's BWE (`enable_bwe`, TWCC/REMB estimator), stashes each `Event::EgressBitrateEstimate` on a shared atomic, and `take_bitrate` relays changes up the reverse channel. `encoder_base::emit_packets` now returns `EmitFeedback { force_keyframe, bitrate_bps }`; `Av1Enc` retargets by rebuilding the rav1e context at the new bitrate (rav1e fixes the rate at construction), hysteresis-gated at 20% so a jittery estimate does not thrash (each rebuild costs a keyframe). Owed: VP8/VP9 runtime bitrate (the `vpx-encode` 0.6 wrapper exposes no `vpx_codec_enc_config_set`); Opus bitrate adaptation; `Bitrate` relay through an intervening transform.
- M243: WebRTC keyframe-request (PLI) plumbing (parity track T2). New reverse-channel signal `Reconfigure::ForceKeyframe` + `AsyncElement::take_reconfigure`, polled by the runners alongside `take_qos` (the GStreamer `GstForceKeyUnit` upstream-event analog). Egress: `WebRtcSink` flags a remote PLI (`Event::KeyframeRequest`) on a shared atomic set by its session task; `take_reconfigure` forwards a `ForceKeyframe` up the incoming link, where the upstream encoder observes it as `PushOutcome::Reconfigure(ForceKeyframe)`. `encoder_base::emit_packets` reports it; `Av1Enc` (rav1e) honors it by overriding the next frame to `FrameTypeOverride::Key`. Ingress: `WebRtcWhepSrc` asks the remote for a keyframe (str0m `StreamRx::request_keyframe`, PLI) on a 1 s interval until the first keyframe arrives, so a mid-GOP join recovers without waiting for the next natural IDR. Owed: VP8/VP9 force-keyframe (the `vpx-encode` 0.6 wrapper does not expose `VPX_EFLAG_FORCE_KF`); `ForceKeyframe` relay through an intervening transform (encoder-adjacent-to-sink works today, same staging as QoS relay).
- M242: WebRTC TURN relay + Opus ingest. A hand-rolled TURN client (`turn.rs`, RFC 5766/8656: Allocate with long-term-credential auth, Send/Data indications, CreatePermission, periodic Refresh) gives `WebRtcSink` + `WebRtcWhepSrc` a `with_turn_server(server, user, pass)` builder and `turn-server`/`turn-user`/`turn-pass` properties, the relay fallback for symmetric NAT / restrictive firewalls a STUN server-reflexive candidate cannot traverse. str0m only offers `Candidate::relayed`; the data plane is the run loop's job: a relayed candidate's transmits all carry `source == relay_addr`, so the loop wraps those in Send indications to the TURN server and unwraps inbound Data indications back into str0m, leaving direct host/srflx paths untouched. Allocation failure degrades to host/srflx. `WebRtcWhepSrc` also gains Opus ingest (`media=audio` / `.audio()`), the receive-side mirror of M239's egress Opus: it offers a recv-only Opus m-line and forwards each packet as `Caps::Audio { Opus }`. Compile-validated against str0m 0.20; the live relay + Opus subscribe paths are owed (sandbox blocks the ports). Remaining: IPv6 reflexive, TURN channel binding, A/V over one PeerConnection.
- M241: WebRTC ICE STUN candidate gathering (`with_stun_server` / `stun-server` property on `WebRtcSink` + `WebRtcWhepSrc`). A minimal RFC 5389 STUN Binding exchange on the ICE socket discovers the server-reflexive (public) address and adds it as a `server_reflexive` ICE candidate, so the elements reach a cloud SFU (LiveKit Cloud, Cloudflare, Dolby Millicast, …) across NAT, not just a same-LAN server. Shared in `webrtc_util::add_ice_candidates`; STUN failures degrade gracefully to host-only. IPv4 v1; TURN relay (symmetric NAT) and IPv6 reflexive remain.
- M240b: g2g↔g2g WHIP→WHEP loopback smoke test (`webrtc_whip_to_whep_loopback`, ignored) - publishes a fixture via `WebRtcSink` and subscribes the same mediamtx stream via `WebRtcWhepSrc` concurrently (both are HTTP clients, so the server relays), asserting the subscriber receives frames in Rust with no browser.
- M240: native WebRTC ingest (`WebRtcWhepSrc`, `webrtc` feature) - the receive-side inverse of `WebRtcSink`. Subscribes to a WHEP endpoint (offer a recv-only H.264 m-line, POST the SDP, apply the answer) and drives str0m's loop on a tokio `UdpSocket`, forwarding each `Event::MediaData` as an Annex-B `DataFrame` (str0m's H.264 depayloader emits start-code framing, g2g's convention) with its RTP-clock PTS. `location` / `bearer` properties + launch name `webrtcsrc`; `with_frame_limit` for bounded runs. The ICE/SDP helpers are now shared with the sink (`webrtc_util`). Compile-validated against str0m 0.20; v1 is video-only (Opus ingest is a follow-up); live subscribe owed.
- M239: `WebRtcSink` Opus audio + a WHIP validation harness. The sink now carries either an H.264 video m-line (90 kHz) or an Opus audio m-line (48 kHz), chosen from the configured caps (`Track`); the session task picks the codec's payload type and RTP clock accordingly. Added `g2g-plugins/examples/whep-player.html` (a standalone WHEP player) and an ignored `webrtc_whip_smoke` test (`FileSrc(h264) -> H264Parse -> WebRtcSink`, driven by `G2G_WHIP_URL` + `G2G_H264_FIXTURE`) with a mediamtx loopback recipe. Simultaneous A/V over one PeerConnection (a `MultiInputElement`) and non-stereo / non-48 kHz Opus remain follow-ups.
- M238: native WebRTC egress (`WebRtcSink`, `webrtc` feature) - publishes a pipeline's H.264 to a WHIP server over a PeerConnection built on the sans-IO `str0m` stack (ICE / DTLS / SRTP), using str0m's pure-Rust `rust-crypto` backend (no OpenSSL system dep). g2g owns the UDP socket + timer and drives str0m's `poll_output` / `handle_input` loop on a spawned tokio task; the element hands access units over a bounded channel (so it stays `Send`, never touching the `Rtc`). WHIP offer/answer over reqwest; `location` / `bearer` properties + launch name `webrtcsink`. Compile-validated against str0m 0.20; the live WHIP publish + browser playback path is owed (sandbox blocks the ports).
- M237: ffmpeg VAAPI hwaccel decode (`FfmpegVideoDec` `Backend::Vaapi`) - attaches an `AV_HWDEVICE_TYPE_VAAPI` device + an `AV_PIX_FMT_VAAPI` `get_format` hook to the generic decoder, decodes on the GPU, then `av_hwframe_transfer_data`s the surface into system memory (NV12) and packs it like the software path. The Linux AMD / Intel hardware path: uses libavcodec's hwframe pool instead of GBM, so it works on Mesa `radeonsi` where cros-codecs `VaapiH264Dec` is blocked. Hardware-validated on an AMD Rembrandt 680M iGPU (`renderD128`). Registered for `gst-launch` as `ffmpegvaapidec` (the gst `vaapih264dec`/`vah264dec` names resolve to it, ahead of cros-codecs `vaapidec`); the new `device` / `output-format` properties pin the render node and pixel layout from a pipeline string (`ffmpegvaapidec device=/dev/dri/renderD128`).
- M236: Python bindings (`g2g-pyapi`, PyO3) - the inverse of `g2g-python`: drive whole pipelines from Python. `g2g.Pipeline(desc)` parses + runs on a background thread, `g2g.AppSrc(channel).push(bytes, pts)` and `g2g.AppSink(channel).pull() -> (bytes, pts) | None` feed/drain it, `bus_poll`/`wait` watch it. Built on the same DSL + appsrc/appsink waist as the C ABI; behind the `python` feature (empty otherwise, so `cargo check --workspace` needs no libpython).
- M235: `appsink` pull mode + zero-copy held samples - alongside the M233 callback, `register_appsink_pull` / `g2g_appsink_new` let the application pull whole frames (blocking `g2g_appsink_pull` or non-blocking `g2g_appsink_try_pull`). A pulled `G2gSample` owns its frame, so its bytes stay valid (zero-copy, including an `appsrc` lend) until `g2g_sample_free`; a full pull queue backpressures the pipeline. Proven: a lend pushed in is pulled out at the same pointer, end to end.
- M234: zero-copy `appsrc` lend - `SystemSlice::from_foreign` wraps an application buffer (raw ptr + len + free callback) with no copy, hidden inside `MemoryDomain::System` so every consumer works unchanged; the free callback fires once when the frame drops. `g2g_appsrc_push_lend` exposes it. A mutating downstream element transparently copies the bytes out first (CoW), keeping the lend read-only. Proven zero-copy in a test: the `appsink` sees the very pointer that was lent.
- M233: `appsrc` / `appsink` elements (application buffer I/O) + C API (`g2g_appsrc_new`/`push`/`end_of_stream`/`free`, `g2g_appsink_set_callback`). The app feeds buffers into a running pipeline and receives them via a `new-sample`-style callback, by name through the launch DSL; the element<->app hand-off is a bounded runtime channel (cross-thread waker, like the Python host). v1 copies bytes across the boundary; zero-copy lend is a follow-up.
- M232: C ABI (`g2g-capi` crate, cdylib/staticlib + `include/g2g.h`) - the language-neutral waist over the launch DSL: `g2g_pipeline_launch` (parse + run on a background thread), `g2g_pipeline_bus_poll`, `g2g_pipeline_wait`, `g2g_pipeline_free`. Built on `g2g-core/multi-thread` so the parsed `'static` graph is `Send`. First slice of the cross-language story (Python/C#/Swift via the C header); `appsrc`/`appsink` buffer I/O is the next slice.
